# DB 操作ルール

## ORM / クエリビルダ

- ORM には Prisma を使用する
- 生 SQL は原則使用しない。パフォーマンス上必要な場合のみ `$queryRaw` を使用する
- マイグレーションファイルは手動で編集しない

## スキーマ設計

- テーブル名は snake_case の複数形にする（例: `user_profiles`）
- カラム名は snake_case にする
- 主キーには UUID（v7）を使用する
- 全テーブルに `created_at`, `updated_at` カラムを設ける
- 論理削除には `deleted_at` カラムを使用する

## クエリ

- N+1 問題を防ぐため、関連データは `include` / `select` で事前にロードする
- 大量データの取得にはページネーションを必ず適用する
- トランザクションは `$transaction` を使用する
- `SELECT *` は使用せず、必要なカラムのみ `select` で指定する

## インデックス

- 検索・ソートに頻繁に使用するカラムにはインデックスを設定する
- 複合インデックスはカーディナリティの高いカラムを先に配置する
- 不要なインデックスは作成しない

## セキュリティ

- ユーザー入力を直接クエリに含めない（パラメータ化クエリを使用する）
- 機密データ（パスワードなど）はハッシュ化して保存する
- DB 接続情報は環境変数で管理する

## バックアップ・運用

- マイグレーションは必ずレビューを通す
- 破壊的変更（カラム削除など）は段階的に行う
