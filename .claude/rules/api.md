# API 設計ルール

## エンドポイント設計

- RESTful な命名規則に従う（リソース名は複数形の名詞）
- URL にはケバブケースを使用する（例: `/user-profiles`）
- ネストは2階層までに制限する（例: `/users/:id/posts` まで）
- バージョニングは URL パスで行う（例: `/api/v1/`）

## リクエスト・レスポンス

- リクエストボディとレスポンスは JSON 形式を使用する
- レスポンスは統一されたエンベロープ形式にする:

  ```json
  {
    "data": {},
    "meta": {},
    "errors": []
  }

  ```

- ページネーションにはカーソルベースを採用する
- 日時は ISO 8601 形式（UTC）で返す

## HTTP ステータスコード

- 200: 成功（GET, PUT, PATCH）
- 201: リソース作成成功（POST）
- 204: 成功（レスポンスボディなし、DELETE）
- 400: バリデーションエラー
- 401: 認証エラー
- 403: 認可エラー
- 404: リソース未発見
- 409: コンフリクト
- 500: サーバー内部エラー

## バリデーション

- 入力バリデーションはコントローラー層で行う
- スキーマバリデーションライブラリ（Zod など）を使用する
- エラーメッセージはクライアントが理解しやすい形式で返す

## セキュリティ

- 認証には Bearer トークンを使用する
- レートリミットを設定する
- CORS を適切に設定する
- 機密情報をレスポンスに含めない
